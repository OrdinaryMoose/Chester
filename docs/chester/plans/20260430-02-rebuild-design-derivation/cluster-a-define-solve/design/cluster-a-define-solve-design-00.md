# Cluster A — Define Solve: Design Brief

Sprint: `cluster-a-define-solve` (under master plan `20260430-02-rebuild-design-derivation`)
Date: 2026-04-30
Status: Approved

---

## Goal

design-specify needs three artifacts to generate feasible, acceptable, complete solutions: a constraint envelope, a coverage map, and a resolution criterion. The first two derive from the existing five proof element categories. The third does not — no proof element today says what observable state counts as the problem being solved, so the brief authors acceptance criteria as agent prose at render time with no upstream source, and the proof's closure gate never references the problem statement. Cluster A formalizes the resolution criterion as a typed proof element, settling its naming, shape, schema, source rule, integrity-check semantics, and closure-check contribution, and defines what makes a problem statement well-formed enough to carry resolution criteria at Phase 4b opening.

---

## Necessary Conditions

- **NC-01 — The resolution criterion is implemented as a new sixth element category in the proof, alongside the five existing immutable types — not as an extension of any existing type and not as a parallel artifact outside the proof.** Named *Resolve Condition* to pair structurally with *Necessary Condition* (both typed conditions, pluralizing naturally) without the misleading conjugate-logic connotation that "Sufficient Condition" would carry. *Reasoning chain:* the five existing element categories are immutable per master-plan RULE-1; the Resolve Condition has semantically opposite job from a Necessary Condition (one restricts, one certifies exit); its source rule differs from any existing category's (agent-proposes-PM-validates is a new pattern); the master-planning collapse test confirmed only the resolution-claim category does not fold into existing types without semantic blur; the proof's element-type registration is a mechanical pattern that accommodates new categories cleanly. *Collapse test:* without this, the Resolve Condition has only three remaining paths — overload an existing category (RULE-1 violation), parallel artifact (loses structural validation and brief-render integration), or remain as agent-prose Acceptance Criteria (the existing failure mode the cluster is closing). *Rejected alternatives:* extend Necessary Condition with a discriminator field (changes NC contract, per-entry ambiguity); parallel artifact alongside the proof (loses validation, forces brief renderer to merge two source artifacts); continue as agent-prose AC (existing failure mode); name as "Sufficient Condition" for NC/SC visual symmetry (logically misleading — the two element types do opposite jobs, not conjugate jobs).

- **NC-02 — The proof carries an enumerated, PM-ratified list of *Concerns* attached to the problem statement, locked at Solve Stage opening.** Each Concern has a stable ID, a short PM-readable label, and (optionally) a one-sentence description. Agent proposes candidate Concerns from the polished problem statement; PM accepts, rejects, adds, or merges; the locked list anchors all subsequent Resolve Condition coverage. *Reasoning chain:* Resolve Conditions need a structurally-typed anchor to prevent free-string anchor drift (gaming-vector simulation finding); the coverage check at closure needs mechanical aspect-match rather than prose-matching; PM-altitude ratification requires the anchor to be PM-readable, not engineer-readable; the agent-proposes-PM-validates pattern extends naturally to the Concern enumeration step. *Collapse test:* without the Concern construct, problem_anchor degrades to free-string prose, recreates anchor drift, recreates the agent-imagined-material failure mode in a different field, and makes coverage at closure a string-matching exercise against agent prose. *Rejected alternatives:* free-string problem_anchor with no enumerated list (recreates drift); single-anchor-per-RC without enumeration (no canonical list to compare for coverage); leave problem statement as free prose (forecloses mechanical coverage entirely).

- **NC-03 — The Resolve Condition element category carries exactly three fields: `statement` (PM-readable observable outcome), `problem_anchor` (Concern ID), `ratification` (PM sign-off, populated when validated). No other fields.** The omission of a verification-artifact field is deliberate: design-specify generates solutions and owns verification (RULE-3); design = the problem (what), not the implementation (how) (RULE-4); verification mechanics derive downstream in the spec layer through the existing acceptance-criteria machinery and test-skeleton scaffolding. *Reasoning chain:* the agent-proposes-PM-validates source rule requires a structural ratification gate distinct from add-time content; verification mechanics belong in the spec layer; PM-altitude ratification cannot evaluate engineer-altitude verification artifacts; the existing five element categories follow minimal-field discipline. Three fields are necessary and sufficient — one for the claim, one for the coverage anchor, one for the validation gate. *Collapse test:* dropping `statement` leaves no claim; dropping `problem_anchor` breaks coverage tracing; dropping `ratification` removes the validation gate and recreates agent-imagined material; adding a verification-artifact field pushes engineer-altitude content onto PM at ratification, prescribes verification before architecture is chosen (crossing RULE-3), and recreates gaming vectors the altitude correction removed; adding priority/severity/ordering violates the minimal-field discipline. *Rejected alternatives:* `forward_test` as required field (crosses RULE-3, altitude failure); `forward_test` as optional (gaming asymmetry between populated and unpopulated entries); verification-pointer reference (couples proof to verification mechanics prematurely); combine statement and verification into one field (merges PM and engineer altitudes); priority/severity/ordering fields (violates minimal-field discipline).

- **NC-04 — The closure gate verifies coverage at Concern granularity: every Concern in the enumerated list must have at least one ratified Resolve Condition anchored to it OR at least one Rule whose statement covers it.** Closure refuses if any Concern is uncovered by both routes. *Reasoning chain:* coverage map is one of the three artifacts specify requires; mechanical coverage prevents the bundling gaming where one giant Resolve Condition satisfies a single-RC check trivially; some real-world resolution-side content is RULE-shaped (preservation, non-existence) rather than Resolve-Condition-shaped; constraint envelope and coverage map already derive from the existing element types. *Collapse test:* without per-Concern coverage, a single bundle-RC anchored to one Concern satisfies an any-RC-exists check trivially while leaving other Concerns silently uncovered — the PM's ratification of the bundle reproduces gestalt approval; coverage gaming wins. Without Rule union, preservation content gets jammed into Resolve Conditions where it doesn't belong, distorting the proof's grammar. *Rejected alternatives:* coverage at any-RC granularity (bundling gaming); coverage at any-element granularity including Evidence (Evidence is fact, not resolution); RC-only coverage with no Rule union (forces preservation/non-existence into RCs where semantics misfit).

- **NC-05 — Ratification is sequential, per Resolve Condition.** The proof MCP refuses batch ratification calls. The validation walk presents Resolve Conditions one at a time; the PM confirms or rejects each individually. *Reasoning chain:* gestalt brief approval is the existing failure mode (designer confirms the whole brief at one yes/no prompt today); ratification fatigue from gestalt approval reproduces that failure mode in a structured wrapper; PM-altitude attention engagement requires discrete units; the agent-proposes-PM-validates pattern implies a discrete validation event per claim. *Collapse test:* batch ratification recreates gestalt approval — the PM's single yes covers N Resolve Conditions without engagement with each; the structural change is hollow and the original failure mode persists in a different shape. *Rejected alternatives:* batch ratification at brief approval time (recreates failure mode); time-gated ratification (ceremony without engagement); optional sequential mode (gaming default becomes batch).

- **NC-06 — Revision of a Resolve Condition's `statement` or `problem_anchor` field after ratification invalidates the ratification.** The closure gate refuses to count un-revalidated Resolve Conditions. *Reasoning chain:* designer-ratified means the PM validated a specific statement-and-anchor content; content can be revised after ratification; silent persistence of ratification across revision reproduces the agent-imagined-material failure mode (PM's sign-off applies to text the PM did not see); the proof MCP already carries a stale-grounding warning pattern adaptable to ratification. *Collapse test:* without the stale-ratification check, the agent revises a ratified RC's statement and the ratification field carries forward unchanged — the PM's sign-off applies to content the PM never approved; the validation gate is bypassed by edit. *Rejected alternatives:* no stale check (reproduces failure mode); semantic-diff threshold for substantive-vs-cosmetic revision (engineer-altitude — PM doesn't reason about diff thresholds; any revision invalidates uniformly); ratification persists across all revisions (loses validity guarantee entirely).

## Rules

- The five existing proof element types (Necessary Condition, Evidence, Rule, Permission, Risk) are immutable for this sprint. Their schemas, source rules, and required fields cannot change. *(Master-plan RULE-1.)*
- Whatever shape is chosen for the resolution-criterion element must carry all five locked attributes: observable, designer-ratified, problem-statement-anchored, forward-looking, non-restrictive. *(Master-plan RULE-7.)*
- design-specify generates solutions. The proof produces input for specify; specify owns architectural shape choice. Resolution criteria are input constraints for specify, not architectural decisions. *(Master-plan RULE-3.)*
- Design = the problem (what), not the implementation (how). The Resolve Condition element captures what state counts as solved, not how the solution achieves it. *(Master-plan RULE-5.)*
- Designer-ratified means the agent authors the proposal and the designer validates it. Distinct from designer-sourced (where the designer originates the content, as with Rules and Permissions). Resolve Conditions follow the agent-proposes-designer-validates pattern. *(Designer-direct, this session.)*
- Constraint envelope and coverage map are derivable from the existing five element types — no new design work for them. Cluster A's substantive design work focuses on the Resolve Condition alone. *(Designer-direct, this session.)*

## Industry Context

- **Hoare logic post-conditions** — formal-methods tradition placing the resolution claim at the structurally precise end of a triple `{P} S {Q}`. Forward-looking, terminal, distinct from invariants. *Pitfalls:* author-authored without stakeholder gate; partial correctness makes no termination claim; weak post-conditions vacuously satisfiable; doesn't fit design-time use cases where command S is not yet specified. *Source:* https://softwarefoundations.cis.upenn.edu/plf-current/Hoare.html
- **NASA / ISO validation criteria** — systems-engineering tradition separating verification ("built it right") from validation ("built the right thing"). Validation requires stakeholder ratification structurally; tracked through Requirements Traceability Matrices. *Pitfalls:* one-time-gate failure when re-validation isn't ongoing; conflation with internal verification; provenance not enforced when stakeholder and engineer are the same person. *Source:* https://swehb.nasa.gov/spaces/7150/pages/16449673/SWE-055
- **BDD / ATDD acceptance scenarios** — Given-When-Then format with structural stakeholder ratification (acceptance-test-driven development requires business-developer-tester agreement before implementation). *Pitfalls:* automation-without-collaboration drift (~25% of teams report business stakeholders disengage); criteria authored after development begins lose forward-looking property; non-deterministic outcomes break crisp Then clauses. *Source:* https://gojko.net/2020/03/17/sbe-10-years.html
- **KAOS achievement goals** — goal-oriented requirements engineering distinguishing achievement (reach a state) from maintain/avoid (restrict a state). *Pitfalls:* refinement-to-constraint conflation loses non-restrictiveness; softgoals lack formal satisfaction criteria; large goal models become impractical to trace. *Source:* https://www.utdallas.edu/~chung/SYSM6309/KAOS-AORE.pdf
- **Goal-Question-Metric (GQM)** — Basili's framework connecting abstract goals to measurable metrics through three levels (conceptual / operational / quantitative). *Pitfalls:* easy-to-measure substituting for what-matters; over-complexity; viewpoint underspecified leading to interpretation drift. *Source:* https://www.cs.umd.edu/users/mvz/handouts/gqm.pdf
- **Definition of Done vs Acceptance Criteria (Scrum)** — DoD as universal team-owned quality floor; AC as story-specific PO-owned conditions. *Pitfalls:* AC written after development begins; DoD/AC drift; non-stakeholder authorship. *Source:* https://www.scrum.org/resources/blog/what-difference-between-definition-done-and-acceptance-criteria

**Cross-pattern synthesis.** Four structural properties recur across all traditions corresponding to a resolution claim: forward-looking-and-terminal, observable-and-stakeholder-facing, non-restrictive (separated from constraint class), anchored-to-intent. These match the master plan's five locked attributes for the Resolve Condition. The sharpest divergence in the field is on **who ratifies** — Hoare/GQM are author-authored without stakeholder gate; BDD/ATDD/NASA make stakeholder ratification structural. The Resolve Condition's source rule (agent-proposes-PM-validates) sides with the structural-ratification camp.

## Risks

- **PM enumerates Concerns sparsely.** Coverage check passes trivially when only two Concerns exist for a problem with six. Agent's candidate-proposal step partially mitigates; ultimately depends on PM judgment.
- **Statement vagueness passes ratification.** A vague observable outcome ("the system resolves the issue") can pass a tired-Tuesday ratification. Sequential per-RC ratification helps but is not bulletproof.
- **Cross-Concern leakage.** A Resolve Condition anchored to Concern-1 actually addresses Concern-3's pain. Anchor field is mechanical (Concern ID match) but semantic correctness is not lint-checked. Designer ratification is the only catch.
- **Behavioral resolution claims with no tooling.** Some Resolve Conditions describe behavioral outcomes whose verification depends on trace tooling (session JSONL, audit log) that may not exist yet. Verification is downstream (spec layer) but if the spec layer can't reach a real artifact, the test skeleton becomes vapor.
- **Brief render and design-specify integration is downstream cluster work.** The Resolve Condition shape lands here; the brief template and design-specify's brief-reading code need updates that ship with cluster B or as a follow-on.
- **Cluster B coordination on transition mechanism.** Resolve Conditions transit Phase 4a → Phase 4b through whatever mechanism cluster B settles. If cluster B's mechanism doesn't carry ratification state cleanly, the validation gate breaks at the transition.

## Acceptance Criteria

- The proof MCP source registers a sixth element type named `RESOLVE_CONDITION` (or equivalent canonical identifier) with the three required fields (`statement`, `problem_anchor`, `ratification`).
- The proof MCP source maintains a Concern construct attached to the problem statement, with operations to enumerate, ratify, lock, and revise Concerns.
- The proof's closure-check function verifies per-Concern coverage with the union rule (ratified Resolve Condition anchored to Concern OR Rule whose statement covers Concern); refuses closure when any Concern is uncovered.
- The proof's ratification API accepts ratification calls one Resolve Condition at a time and refuses batch shapes.
- Revision of a Resolve Condition's `statement` or `problem_anchor` invalidates its ratification automatically.
- The brief template's Acceptance Criteria section is replaced (or supplemented) by a Resolve Conditions section that renders from structurally-typed proof elements, each entry carrying its statement, anchor (Concern ID + label), and ratification quote.
- The brief template's structure includes a Concerns section listing the locked enumeration with IDs and labels.
- Existing five element categories (NC, Evidence, Rule, Permission, Risk) function unchanged after the sixth-type addition.
- design-specify continues to read briefs that follow the new template without breaking; AC-{N.M} entries in the spec derive from Resolve Condition statements.
- Tests cover: sixth-type registration; Concern enumeration and locking; per-Concern coverage check (positive and negative cases); sequential ratification refusal of batch calls; stale-ratification on statement/anchor revision.
