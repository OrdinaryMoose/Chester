# Design Brief: Chester Decision-Record / Constraint-Triangle System

**Sprint:** 20260424-01-build-decision-loop
**Date:** 2026-04-24
**Status:** Approved
**Parent:** `docs/feature-definition/Pending/spec-test-code-decision-loop/`

## Goal

Chester today runs as a one-directional pipeline — design flows into spec, spec into plan, plan into code, code into finish, finish into archive. Nothing flows back. But the work Chester does is inherently a feedback loop: implementation surfaces decisions the upstream artifacts never anticipated, and those decisions need to reach spec, tests, plans, and affected code to preserve coherence. This design adds the structural loop-closure mechanism Chester lacks, addressing the "specification drift" problem the April 2026 industry survey identifies as the unresolved frontier across the agentic-skills ecosystem. Chester becomes pipeline-shaped and loop-shaped at once.

## Necessary Conditions

1. **Decision records must be created prospectively at the moment ambiguity surfaces during write-code — not retrospectively at sprint end.** Prospective timing is what enables propagation to spec, tests, plans, and affected code before task completion. *Collapse test:* records become post-hoc reconstructions with no update path; the system collapses back to today's retrospective-only state and drift returns unaddressed. *Rejected alternatives:* retrospective capture at finish-write-records only; per-commit capture regardless of ambiguity; manual designer-triggered capture.

2. **Spec-level and plan-level discoveries must be discriminated mechanically, not by LLM judgment alone.** *Collapse test:* system either over-fires (every commit spawns a record, ceremony explodes) or under-fires (spec-level discoveries silently resolved as plan-level). *Rejected alternatives:* LLM-judgment-only; treat all as spec-level; treat all as plan-level; designer-judges-each-time manual classification.

3. **Every decision record must carry the full field set defined in the record-structure rule** — header (ID, Sprint, Task, Status, Tags) and body (Trigger, Context, Options Considered, Chosen, Rationale, Spec Update, Test, Code, Supersedes, Superseded By); Status carries one of Active, Superseded, or Abandoned. *Collapse test:* records decay to changelog entries; four-way traceability breaks; "why does X behave this way?" six months later has no answer — the failure mode the design exists to prevent.

4. **Propagation from a decision record to spec, tests, plans, and affected code must be a blocking gate on task completion — advisory propagation is insufficient.** *Collapse test:* propagation becomes advisory; within weeks, records exist with stale spec and tests alongside them; drift is functionally identical to today. *Rejected alternatives:* advisory propagation with reviewer enforcement; retrospective reconciliation at finish-plan only; LLM-enforced probabilistic gate.

5. **Propagation must reach backward into code already committed in earlier tasks of the current sprint** — the "implementation" destination named in the problem statement — not forward propagation only. *Collapse test:* sprint ends with Task 1 code violating updated spec; the living-plan property fails within the sprint itself.

6. **The persistent decision store must live at `/docs/chester/decision-record` as a cross-sprint artifact, separate from per-sprint directories under `plans/` or `working/`.** *Collapse test:* cross-sprint propagation dies; future sprints re-derive what prior sprints decided; outer loop stays broken. *Rejected alternatives:* records stay in `plans/{sprint}/` only; records duplicated in both per-sprint and central locations; records only in `summary/` reasoning-audit; records embedded in the spec itself.

7. **Supersede across sprints must be append-only with explicit bidirectional back-links** — the superseded record is never rewritten or deleted. *Collapse test:* archaeological chain breaks on first supersede; "why did this change?" becomes unanswerable; supersede degenerates into destructive update. *Rejected alternatives:* rewrite-in-place supersede; soft-delete without preserving forward link; implicit supersede detection (pattern matching) instead of explicit back-links.

8. **`plan-build` must consult the persistent decision store when building each new sprint's plan**, surfacing relevant prior decisions as named inputs that inform spec, plan, and implementation choices of the new sprint. *Collapse test:* records accumulate but never inform anything; outer-loop half of the problem statement goes unaddressed; cross-sprint drift is architecturally unsolved.

9. **The mechanical discriminator between spec-level and plan-level discoveries is test-skeleton pass/fail.** `build-spec` auto-scaffolds a test skeleton for each acceptance criterion at spec time; during write-code, a discovery is spec-level if implementation produces observable behavior no existing skeleton covers, plan-level if implementation fails an existing skeleton (bug, not a decision) or produces behavior already covered (no decision to record). *Collapse test:* no viable fallback — discrimination falls back to LLM judgment (which condition #2 rejects) or designer judgment (which conflicts with the agent-autonomous authority rule). *Rejected alternatives:* contract-boundary check via explicit contract definitions (Chester has none; separate large build); spec-criterion-ID reference by the implementer (recapitulates the judgment problem); LLM-judgment discrimination; designer-judges-each-time manual classification.

10. **Decision records are stored direct at `/docs/chester/decision-record` from the moment of capture during write-code; no sprint-close promotion, distillation, or curation step.** `plan-build` applies a filter at read time (sprint subject match, tag, supersede-active-only, status = Active, or recency) rather than reading the entire store. *Collapse test:* system falls back to one of the failing positions — sprint-scoped-then-promoted (classic ADR failure), dual-write (sync cost, divergence risk), or curated distillation (ADR-grade curation ceremony); all three are worse than the current position. *Rejected alternatives:* sprint-scoped then promoted at sprint close; dual-write with sprint-scoped and cross-sprint copies; curated distillation at sprint close; read-whole-file at plan-build time.

## Rules

- **Persistent store location** — The persistent memory artifact lives at `/docs/chester/decision-record` as a flat markdown file, similar in form to `memory.md`. Separate from per-sprint artifacts; sibling to `docs/chester/plans/` and `docs/chester/working/`.
- **Prior-art anchor** — Design anchors on dbreunig's SDD Triangle, gotalab cc-sdd spec-as-contract, and Chester's own architecture table from the context-handover doc; does not re-derive from external design-rationale history.
- **Problem statement** — Chester runs pipeline-shaped but works loop-shaped: implementation-surfaced decisions never propagate back to spec, tests, plans, and implementation, within the current sprint or for future sprints. Four destinations, two altitudes.
- **Two altitudes in scope** — Within-sprint loop closure (decision record updates spec, tests, remaining plan tasks, and earlier-task code before task completion) and cross-sprint propagation (decisions reach into future sprints' plan-building) are layered lifecycle phases of the same atom, not separate features.
- **ID format** — Decision record identifiers follow `YYYYMMDD-XXXXX` format where `XXXXX` is a zero-padded per-day sequence. Identifier doubles as creation date — no separate Date field.
- **Field set mandatory** — Every record carries the full field set: Header (ID, Sprint, Task, Status [Active / Superseded / Abandoned], Tags); Body (Trigger, Context, Options Considered, Chosen, Rationale, Spec Update, Test [name + commit SHA], Code [file:line + commit SHA], Supersedes, Superseded By). Some fields may carry "none" with justification; none may be omitted. Status becomes 'Superseded' when a later record explicitly supersedes; 'Abandoned' when the sprint that produced the record is abandoned without merge; otherwise 'Active'.
- **Agent-autonomous capture** — Records are captured by the implementing agent autonomously when the trigger criterion fires (two or more valid implementations exist and existing artifacts don't determine). No human-approval gate intercepts record creation; the designer reviews records during standard review flow. Agent-autonomous capture preserves the non-interrupt discipline of the structured-decision approach.

## Evidence

- Chester runs as a one-directional pipeline: design → spec → plan → execute → finish → archive. Each stage feeds the next; no stage reads back into upstream artifacts during normal operation. (`skills/*/SKILL.md`, `hooks/hooks.json`)
- `finish-write-records` captures decisions retrospectively by parsing the session JSONL transcript at sprint end. The reasoning-audit artifact lands in `summary/` and does not update spec or tests. (`skills/finish-write-records/SKILL.md`)
- The design-proof MCP defines an append-only element grammar of EVIDENCE, RULE, PERMISSION, NECESSARY_CONDITION, RISK with grounding links and supersede semantics — usable as the decision-record vocabulary. (`skills/design-large-task/proof-mcp/`)
- `execute-write` tracks BASE_SHA and HEAD_SHA per task in session state but does not persist the task-to-commit mapping to any artifact. The raw material for traceability exists; the persisted artifact does not. (`skills/execute-write/SKILL.md`)
- `plan-build` invokes `plan-attack` (mandatory) and `plan-smell` (conditional) as the hardening gate. These are the closest existing analog in the field to spec-quality pressure-testing. (`skills/plan-build/SKILL.md`, `skills/plan-attack/`, `skills/plan-smell/`)
- Chester's plugin hook surface is SessionStart only. No PreToolUse, PostToolUse, PreCommit, or per-task hooks. Recently-reverted compaction hooks showed ambient hook infrastructure in Chester depends on platform surfaces that can shift. (`hooks/hooks.json`)
- dbreunig's SDD Triangle (March 2026) identifies the structural diagnosis: SDD is a feedback loop, not a one-directional pipeline. Spec, tests, and code are three nodes whose coherence is maintained by continuous reconciliation.
- gotalab cc-sdd establishes spec-as-contract: spec owns observable behavior at component boundaries; implementation detail within boundaries belongs to code. This is the discipline that separates spec-level from plan-level discovery.
- Decision-record trigger criterion: a record is created precisely when two or more valid implementations exist and the existing spec/plan/tests do not determine which to choose. Not per-commit, not per-task — only when ambiguity surfaces. (chat dump, `docs/feature-definition/Pending/spec-test-code-decision-loop/`)
- Decision records are append-only and supersede-capable. When a later sprint revisits a behavior, a new record references and supersedes the earlier one; neither is deleted. The chain is the archaeological record of how behavior evolved. (chat dump)
- Validation asymmetry across the three triangle relationships: Code↔Test is deterministic (tests pass or fail). Spec↔Test is structural (traceability, code-parseable, not LLM). Spec↔Code is probabilistic (LLM pressure-test only; the deterministic gate is the decision log itself). (chat dump)
- Within a single sprint, spec and plan form a hierarchy: spec is above plan. Spec-level discoveries propagate down and force plan changes; plan-level discoveries stay local and do not touch spec. (chat dump)
- Four-way bidirectional traceability: from spec clause (what decisions contributed), from test (what decision this validates), from code (what decisions ground this function), from decision (what spec/tests/code this touches). This is the queryability discipline that makes decision records load-bearing rather than decorative. (chat dump)
- A pending feature-definition brief (solution-design-language-kb, 2026-04-23) proposes a persistent proof-grammar-structured knowledge layer. Currently at proposal / phase-0 research stage, not implemented. (`docs/feature-definition/Pending/solution-design-language-kb-00.md`)
- Spec Kit (`/speckit.reconcile`) and OpenSpec (`/opsx:sync`) shipped drift-reconciliation commands since the April 2026 survey. Retrospective reconciliation at commit time is the current shipped competitor shape.
- Design-rationale capture systems (IBIS / gIBIS / QOC) have a 35-year adoption-failure record rooted in asymmetric cost — capture is expensive at author-time, benefit is deferred. LLM-in-the-loop may change the calculus but unproven.

## Industry Context

- **Nygard ADR (2011)** — canonical baseline for architectural decision records. Append-only with supersede semantics; fields are Title, Status, Context, Decision, Consequences. Pitfalls: records retrofitted as justification rather than documentation; ~10-page records go unread; no mechanism to enforce creation at decision time; curator roles atrophy. Source: [Cognitect: Documenting Architecture Decisions](https://www.cognitect.com/blog/2011/11/15/documenting-architecture-decisions).
- **MADR (Markdown ADR)** — Nygard evolution adding explicit Options Considered, Decision Drivers, per-option pros/cons. Tooling includes adr-tools, Log4brains, Structurizr. Pitfalls: heavier than Nygard, thin enforcement of bidirectional code/test linkage. Source: [adr.github.io](https://adr.github.io/).
- **dbreunig SDD Triangle (March 2026)** — direct prior sibling: spec/tests/code as three nodes maintained by continuous reconciliation, with "Plumb" tool extracting decisions at commit time for human approval. Same three-node framing, same feedback-loop diagnosis. Pitfalls: same asymmetric-cost concern as the broader ADR tradition; reconciliation is retrospective (commit-time) rather than prospective (discovery-time). Source: [dbreunig: The Spec-Driven Development Triangle](https://www.dbreunig.com/2026/03/04/the-spec-driven-development-triangle.html).
- **gotalab cc-sdd spec-as-contract** — spec owns observable boundaries; implementation fills interior. Establishes the altitude discriminator that separates spec-level from plan-level discovery. Pitfall: the tooling to enforce the contract boundary is thin. Source: gotalab cc-sdd (Kiro-derived Claude Code framework).
- **Spec Kit `/speckit.reconcile` and OpenSpec `/opsx:sync`** — 2026 shipped drift-reconciliation commands: after implementation, compare spec against code, surface divergence, human picks which side updates. Pitfalls: retrospective not prospective; reconciliation at commit time rather than at discovery time; no mechanism for retroactive propagation into earlier-task code. Source: [Spec Kit vs BMAD vs OpenSpec (2026)](https://dev.to/willtorber/spec-kit-vs-bmad-vs-openspec-choosing-an-sdd-framework-in-2026-d3j).
- **Design rationale tradition (IBIS / gIBIS / QOC)** — 35-year record of structured capture systems failing due to asymmetric cost. Pitfalls: capture is expensive at author time, benefit is deferred and diffuse; curator roles atrophy; records retrofitted or ignored. Root cause identified in the literature: no designer can be expected to enter quality rationale solely for the possible benefit of a possibly unknown future reader. This design answers the asymmetry via the trigger criterion (fires only on actual ambiguity), LLM-assisted drafting (lowers author-time cost), agent-autonomous capture (no context switch), and direct-store (no curation ceremony). Source: [Hypermedia support for argumentation-based rationale (gIBIS/QOC retrospective)](https://oro.open.ac.uk/3032/1/HypermediaSupport.pdf).
- **LangGraph `interrupt()` pattern** — AI-agent-native primitive for in-flight human-in-the-loop decision capture. State checkpointed, execution pauses, payload shown to user, resumes cleanly. Pitfall: focuses on pause/resume rather than structured artifact production; would conflict with the agent-autonomous authority rule if adopted. Source: [LangChain: making HITL agents with interrupt](https://www.langchain.com/blog/making-it-easier-to-build-human-in-the-loop-agents-with-interrupt).

## Risks

- **Flat-file scale collapse at `/docs/chester/decision-record`** — the persistent store grows unboundedly as records accumulate across sprints; past some threshold (hundreds of entries) the file becomes unreadable or token-prohibitive for `plan-build` consumption. Designer accepts this risk for now and will revisit when it grows. Mitigations available (not yet adopted): shard by topic, shard by year, hard-cap with rotation, search-index instead of flat-file.
- **Abandoned-sprint records persist in the cross-sprint store** — when a sprint is abandoned (branch discarded, worktree closed without merge), records it wrote to `/docs/chester/decision-record` remain. Mitigated by the `Status = Abandoned` value (per the field-set rule), which `plan-build`'s read-time filter excludes from consideration alongside Superseded records. Records are preserved as historical artifact (real decisions at real moments) but do not pollute plan-building input.

## Acceptance Criteria

- Decision records are created when the trigger criterion fires during write-code (two or more valid implementations exist and existing artifacts don't determine), and not otherwise.
- Records live at `/docs/chester/decision-record` and follow the `YYYYMMDD-XXXXX` ID format.
- Every record carries the full mandatory field set; field-set validation passes for all records at commit time.
- A task is not marked complete unless spec, tests, plans, and affected code align with any decision records created during that task.
- When a spec-level record is created mid-sprint, code committed in earlier tasks of the current sprint is re-checked against the updated spec clause and revised if required.
- Test skeletons are auto-scaffolded at `build-spec` time, one per acceptance criterion, and serve as the mechanical spec-level/plan-level discriminator during write-code.
- A record carrying `Status: Superseded` has an explicit forward-link to its successor; its successor has a back-link to it; neither is rewritten or deleted.
- A record whose sprint is abandoned carries `Status: Abandoned` and is excluded from `plan-build`'s read-time filter.
- `plan-build` reads the decision store with a filter (sprint subject match, tag, supersede-active-only, status = Active, or recency) and surfaces matching records as named inputs when building each new sprint's plan.
