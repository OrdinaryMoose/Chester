# Design Brief — Decision Record System

**Sprint:** `20260501-01-fix-decision-record`
**Date:** 2026-05-01
**Status:** Approved (round 7 ratification)

## Goal

Capture agent-made decisions across the design-specify, plan-build, and execute-write cycle in a simple searchable record format that preserves knowledge for future-me and informs future agent-level planning. The existing decision-record system was built at the wrong altitude — its trigger was structurally inverse-coupled to good upstream design and never fired in production. The replacement piggybacks on the reasoning-audit pass that already discriminates substantive decisions from the session JSONL transcript, emitting structured records to a single cross-sprint markdown corpus alongside the audit prose.

## Necessary Conditions

- **NC-01 — Records are emitted at finish-write-records during the reasoning-audit pass, derived from the same JSONL transcript source the audit reads.** *Reasoning chain.* The audit already discriminates substantive decisions from a complete-session JSONL source; capture must avoid TDD-loop participation; capture must span the full session cycle including upstream design work; records must not use an MCP. Piggybacking emission on the audit pass uses the existing validated discrimination, fires outside the build cadence, naturally covers all stages in one pass, and writes via direct file I/O at finish time. *Collapse test.* If records are emitted earlier in the cycle, capture either fragments across multiple skill points, requires agent memory rather than transcript source, costs more tokens per decision, or duplicates and fights the audit's already-validated discrimination logic. *Rejected alternatives.* Per-decision emission at each substantive moment in design-specify and plan-build — rejected because it requires agent memory or new skill steps at every decision point, costs tokens proportional to decision count, and duplicates the audit's discrimination judgment without the transcript source. Stage-close emission at end of design-specify and plan-build — rejected because it fragments capture across skill points, partial coverage compared to the audit's whole-session reach (which already includes design-large-task / design-small-task), and relies on session memory rather than authoritative JSONL transcript.

- **NC-02 — Records and audit entries are parallel outputs over the same JSONL source with independent discrimination filters; a decision included in one need not be included in the other.** *Reasoning chain.* The audit serves session-internal recall; records serve cross-sprint trend-finding. The two altitudes have different relevance criteria — a minor decision can be session-noise yet cross-sprint-precedent. The filters must be decoupled; coupling them strips the freedom to tune each surface independently and forces records to inherit the audit's session-altitude discrimination. *Collapse test.* If records strictly mirror audit selections (filters coupled), small precedent-setting decisions that aren't session-recall-relevant are lost from the cross-sprint corpus, and the system cannot tune cross-sprint relevance separately from session-recall relevance.

- **NC-03 — Records live as appended entries in a single markdown file at `docs/chester/decision-record/decision-record.md`; new records append, prior records are never modified.** *Reasoning chain.* The corpus is a single queryable cross-sprint surface; append-only is invariant; the existing path is established by prior design. A single file at the canonical path supports grep-based query without directory traversal, preserves all priors intact, and matches the established convention. *Collapse test.* Per-file-per-record breaks the single grep-able corpus property and forces directory-listing conventions. Mutating priors to mark supersession breaks the append-only invariant and loses the historical decision trail.

- **NC-04 — Each record carries structured fields enabling agent query via grep / parse without LLM judgment — minimum: id, date, sprint, stage, title, decision, rationale, alternatives, tags, supersedes, artifact_refs.** *Reasoning chain.* Future agents query the corpus mechanically (grep, awk, jq-style frontmatter parse); the format must remain simple; the audit's existing format already provides most of the substance fields (title, context, alternatives, decision, rationale, confidence). The record field set is the audit format adapted with cross-sprint affordances added — id for stable reference, sprint and stage for filter axes, tags for query axes, supersedes for chain navigation, artifact_refs as cooperation handle to session-level artifacts. *Collapse test.* Without structured fields, query becomes LLM-judgment-mediated parsing across prose; tag drift, missing IDs, and unstructured supersession links make cross-sprint discoverability unreliable.

- **NC-05 — The decision-record-specific additions from the 2026-04-24 build-decision-loop merge are removed surgically; every change that landed after the merge is preserved.** *Reasoning chain.* The design reverts to the pre-04-24 baseline; no MCP is allowed; no TDD-loop participation is allowed; post-04-24 changes carry independently-validated improvements that do not participate in the DR system or its forbidden patterns. The revert is surgical — selective line and file edits — rather than a wholesale merge-revert which would also drop the keep-bucket changes. *Collapse test.* If the revert is non-surgical, every later improvement is also lost — provenance stamping, named-subagent fork policy, pluggable Understanding-MCP, ground-truth automation. If the revert is partial, dangling references appear (`dr_audit` pointing at a removed MCP, `must-remain-green` referencing a non-existent `dr_query`) and the system breaks at integration boundaries.

- **NC-06 — Finish-write-records gains a new emission step that, during the same agent pass producing the reasoning audit, writes structured records to `docs/chester/decision-record/decision-record.md` via direct file append, applying its own discrimination filter independent of the audit's.** *Reasoning chain.* Capture fires at audit time (NC-01); filters are independent (NC-02); records live append-only at the canonical path (NC-03); records carry structured fields (NC-04). The audit-emission must be a concrete step in finish-write-records that performs the file write — without it, the rules and conditions describe a system that has no implementation surface and capture never fires. *Collapse test.* Removing the emission step leaves the rules and conditions in place but no mechanism executes them; the system describes the right capture pattern but never produces a record.

## Rules

- Records are written as markdown in `docs/chester/decision-record/`.
- Capture format is agent-focused — structured machine-queryable fields rather than narrative prose; future agents are the primary readers.
- Store is append-only / continuous-write; existing records are never rewritten.
- Supersession is by reference — a new record references the prior record, the prior record stays intact, and the newer record is authoritative over the older.
- Capture covers agent-made decisions across the design-specify, plan-build, and execute-write cycle — not just one stage.
- Keep the solution simple — minimum viable shape that meets the goal.
- Revert execute-write to its pre-04-24 shape: drop the trigger-check and propagation step, drop the observable-behaviors artifact, drop the spec-reviewer input-restriction module, drop the propagation-procedure and test-generator references; restore implementer + spec-reviewer + quality-reviewer per-task chain.
- Capture serves two altitudes of value: preserving knowledge for future-me (the human reading later) and informing future agent-level planning (the machine querying later).
- The decision-record system will not use an MCP. Capture is implemented via direct file operations at the skill level, not via tool calls to a server.
- The decision-record system works cooperatively alongside session-level artifacts, not as a replacement. Both surfaces can contain duplicate information. Session artifacts serve session-internal recall ("what did this sprint do"); decision records serve cross-sprint trend-finding ("what's been decided about X across sessions") where reading every prior session's artifacts is impractical.
- The decision-record system does not participate in the TDD write loop. Capture is not a gate on test-first / implement / test-pass discipline; execute-write's build cadence runs uninterrupted by record writes.
- Capture scope spans the entire session including design-large-task and design-small-task when run in-session, not just the design-specify, plan-build, execute-write subset.
- Decision records and audit entries are parallel outputs over the same JSONL source. Each surface has an independent discrimination filter — a decision selected by one filter need not be selected by the other. Filters can be tuned independently across the two surfaces.

No permissions granted. All rules apply as stated.

## Industry Context

- **Architecture Decision Records (ADRs).** Self-contained decision documents stored as a queryable corpus per file with stable IDs; enforcement is social at code review. *Pitfalls.* Rot when no one tends the log; retroactive entries lose context; checkbox adoption when discipline weakens. *Sources.* `https://docs.aws.amazon.com/prescriptive-guidance/latest/architectural-decision-records/adr-process.html`, `https://martinfowler.com/bliki/ArchitectureDecisionRecord.html`, `https://microsoft.github.io/code-with-engineering-playbook/design/design-reviews/decision-log/`.
- **Plumb commit-time gate (dbreunig 2026).** Pre-commit hook intercepts and extracts decisions from session conversation, blocks commit until reviewed. *Pitfalls.* Inverse-coupled to spec quality (fires less when specs are good); diff-vs-conversation extraction noise; gate friction at commit time. *Source.* `https://www.dbreunig.com/2026/03/04/the-spec-driven-development-triangle.html`.
- **Spec Kit `/speckit.reconcile` and OpenSpec `/opsx:sync`.** Post-implementation reconciliation commands; advisory rather than blocking. *Pitfalls.* Optional invocation accrues records only when developer perceives a gap; same inverse-coupling problem; conflict-resolution logic underspecified. *Sources.* `https://github.com/github/spec-kit/issues/1063`, `https://github.com/Fission-AI/OpenSpec/blob/main/docs/commands.md`.
- **IBIS / gIBIS / QOC tradition.** Argumentation-based rationale capture as graphs of questions, options, and arguments. *Pitfalls.* Cognitive overhead asymmetry between capturer and consumer; required hypermedia/IBIS literacy; advisory-only systems decay reliably under deadline pressure (35-year history). *Sources.* `https://www.academia.edu/2655542/Hypermedia_support_for_argumentation_based_rationale_15_years_on_from_gIBIS_and_QOC`; `https://link.springer.com/chapter/10.1007/978-3-540-30998-7_3` (Horner & Atwood); `https://mccricks.wordpress.com/2011/12/02/20-years-on-from-gibis-and-qoc/`.
- **Inverse-coupling pattern.** No canonical industry name, but the phenomenon recurs — better upstream quality reduces downstream tool trigger rate. *Pitfalls.* Frequently misread as system failure when it is system success; inflates noise if mitigated by lowering detection threshold. *Sources.* `https://www.jetbrains.com/pages/static-code-analysis-guide/quality-gates-in-software-development/`; `https://www.augmentcode.com/tools/best-spec-driven-development-tools`.
- **Soft-disable / inert-in-place lifecycle.** Mature teams handle non-earning systems through clean removal, hard disable, scope reduction, or reframe rather than silent inertness. *Pitfalls.* Silent inertness is dangerous because the team believes the system is active; configuration drift accumulates. *Sources.* `https://www.statsig.com/perspectives/tips-for-unused-feature-flag-clean-up`; `https://launchdarkly.com/blog/what-are-feature-flags/`.

## Risks

- **Discriminator inheritance.** The audit's discrimination logic (the four-to-twelve non-trivial-decision filter) is the source the records' filter shares. If both filters miss a substantive decision, records miss it permanently. The parallel-filter rule (NC-02) permits a more inclusive records filter, but the records filter is itself agent-judgment-driven and can inherit blind spots at its own altitude.
- **Tag drift without validator.** Agent-written records carry tags as a key query axis (NC-04), but no MCP validates the tag set (RULE: no MCP forbids server-side validation). Across sprints, an agent might write `arch` where a prior sprint wrote `architecture`, breaking grep-based queries. Mitigation is documenting a small canonical tag list inside the writing skill plus existing-tags-first discipline; enforcement is by convention only.

## Concerns

- **CERN-1 — Capture trigger placement**: where in the design-specify, plan-build, and execute-write cycle a decision gets recorded — at end of each stage, mid-stage, on each substantive decision, or only at sprint close.
- **CERN-2 — Record shape**: what fields and structure an agent-queryable record carries.
- **CERN-3 — Cross-sprint discoverability**: how future agents find relevant records when planning a new sprint.
- **CERN-4 — Execute-write revert scope**: what is removed when execute-write reverts to its pre-04-24 shape, and what dependent integrations need to adjust.
