# Sprint-02 Proof Layer — Design Brief

**Sprint:** `20260511-01-mp-redesign-proof-system/sprint-02-proof-layer`
**Date:** 2026-05-13
**Status:** Approved — winning design from four-pole structural-design debate (Purist / E pole)
**Source:** Experimental four-pole Cartesian design debate using agent-teams mechanic; selected by human evaluator through three elimination cycles.

---

## Goal

Sprint-02 implements the **Proof Layer** — the Domain layer in the Chester proof system's three-layer hexagonal architecture (`03-architecture.md` §2). The content of the Domain (schema, mutations, authority, lifecycle, closure policy, integrity policy, friction policy, restructuring, render, counterfactual) is fixed by `05-domain-spec.md`; sprint-02 designs the **structural shape** that realizes that content as an implementable JavaScript codebase. The structure must mechanically realize the architecture's §10 payoffs (Engine reusable, Domain testable, Interface replaceable, counterfactuals free, narrow-port audit adapter clean, Adversary integrable) — by enforcing each promise *structurally at the code level*, not behaviorally at the code-review level. The design must consume the Engine through the six substrate ports specified by ADR-0012 as amended by ADR-0013 (`IFactStore`, `IRuleStore`, `IQueryEngine`, `ISnapshotRestore`, `IExplain`, `ITransaction`), expose the Domain to the Interface through the seven delivery ports specified by Architecture §4.2, and inject the four cross-cutting adapters (`IClock`, `IIDAllocator`, `IConsentVerification`, `IPersistenceRepository`) at session boundary. Target size: ~1,500–2,500 LOC across approximately 11 files plus a bridge, per Domain Spec §13.

## Prior Art

**Master plan cascade.** The Chester proof system master plan `20260511-01-mp-redesign-proof-system` carries a multi-document design cascade: `01-vision.md`, `02-conops.md`, `03-architecture.md`, `04-engine-spec.md`, `05-domain-spec.md`, `06-interface-spec.md`, `07-migration-plan.md`, `08-test-strategy.md`, plus 19 ADRs (current cascade head: ADR-0013, which relocates `IMaterializer` to Domain and specifies transaction-visibility as read-own-writes). Sprint-01 (`sprint-01-proof-backend`) is implementing the Engine layer; sprint-03 (`sprint-03-presentation-layer`) will implement the Interface layer using the Domain delivery ports sprint-02 exposes. The Domain Specification (`05-domain-spec.md`, 800 lines) is the authoritative source for what the Domain does; sprint-02 is the design step that translates spec content into implementation structure.

**Architecture commitments inherited.** Six substrate ports (ADR-0012/0013), seven delivery ports (Architecture §4.2), four cross-cutting adapters (Architecture §4.3), the inward-dependency rule (§3), the forward-solve paradigm (ADR-0007) with approval-as-engine-body-literal (ADR-0003), the closed-set discipline (§11.11), the named-port-discipline forbiddens (§4.5), the canonical mutation flow (§5 — Architecture's load → verify → begin → mutate → check → commit-or-rollback → save pattern), and the typed-citation-via-predicate-signatures move (ADR-0004). The architecture also names load-bearing failure modes (§14): cheap-path closure, closed proof of wrong design, stalled proof, race-to-presentation, authority drift — each with named structural mitigations the Domain must implement.

**Four-pole experimental design debate.** The structural shape was selected via a Cartesian four-pole debate (Conservator / Innovator / Pragmatist / Purist) using the Claude Code agent-teams mechanic with peer-to-peer DMs. Three elimination cycles converged to the Purist design (the verb-as-data + structural-discipline position). Key contestations the winning design survived:

- **Innovator (eliminated cycle 1).** Held a structurally near-identical position to Purist; the verb-as-data + two-registry argument was redundantly defended and one of the pair was eliminated to preserve diversity of remaining positions. Innovator's contribution survives in the Purist design's adoption of declarative `OPERATION_SPECS` and `CATEGORY_REGISTRY` registries.
- **Conservator (eliminated cycle 2).** Held the canonical middle: §13 file names verbatim, verb-as-code-with-combinator, four role-narrowed bundles, two-file schema/translation split. Conservator's deepest critique against Purist: "Rules are bulk-symmetric (OCP-by-data applies); verbs are eight named entities with distinct §6.1 prose (OCP-by-data does *not* extend upward)." Purist addressed this by introducing a typed `customPostCheck` escape hatch for verbs whose post-conditions don't reduce to declarative query patterns — preserving the §6.1-pattern-lives-once property while honestly admitting the irreducible 5% asymmetry.
- **Pragmatist (eliminated cycle 3, head-to-head).** Held the minimum-helpers / debuggability-first stance: verb-as-code in plain JS, accessor-method ISP, single-file `CATEGORY_DEFS`, save at Interface boundary. Pragmatist's sharpest attack: "`customPostCheck` exists because 3 of 8 verbs (37.5%) need post-condition logic that doesn't reduce to declarative query patterns. That isn't 5% asymmetry — it's more than a third of the verbs needing an escape from the data shape." Purist's reply: typed escape hatch at a named location is principled (the asymmetry is enumerated in the type system); the alternative (eight scripted recipes with inline subtle exceptions) disperses the asymmetry across function bodies where drift becomes review-level rather than type-level. Decisive criteria named by the evaluator: (1) maintainability for sole developer 6 months out — type system substitutes for absent peer review; (2) external customer usability for sprint-03 — load/save inside `runOperation` gives one-line MCP adapters versus multi-line scaffolding at every tool handler.

## Scope

**In scope:**

- Module structure for the Domain layer — 11 files plus a bridge file (~2,050 LOC target).
- The `OperationSpec` typed record shape, including the typed `customPostCheck` escape hatch for irreducible verb-shape asymmetry.
- The `CategoryDescriptor` typed record shape for the §2 translation table.
- The four role-narrowed port bundles (`ReadPorts`, `WritePorts`, `ProbePorts`, `FullPorts`) and the rule that every Domain function declares which bundle it consumes by parameter type.
- The `runOperation` pipeline function realizing Domain Spec §6.1 line-by-line.
- The three-phase rule registration model (Phase A: cold-start static rules; Phase B: cold-start element-rule template stratification; Phase C: per-element template instantiation inside the transaction).
- The save-after-commit boundary inside `runOperation`, with `POST_COMMIT_SAVE_FAILED` as a typed Domain error.
- The closed-set discipline made physical in `tags.js`.
- The schema/translation SRP split (`schema.js` for shape validation; `translation.js` for translation generation + rule templates).
- The render module's read-only narrowing via `ReadPorts` parameter type.
- The counterfactual module's snapshot-restore-on-throw invariant via `ProbePorts`.
- The bridge as the single assembly seam for the concrete Engine.
- The seven delivery-port facade methods on the bridge that the Interface layer (sprint-03) consumes.

**Out of scope:**

- **Content of the Domain.** What the Domain does — schema, mutations, authority, lifecycle, closure policy, integrity policy, friction policy, restructuring, render, counterfactual — is fixed by `05-domain-spec.md`. Sprint-02 designs structure, not content.
- **The Engine implementation.** Sprint-01 (`sprint-01-proof-backend`) implements the six substrate ports. Sprint-02 consumes them; it does not implement them.
- **The Interface implementation.** Sprint-03 (`sprint-03-presentation-layer`) implements the MCP tool surface, wire format, and persistence adapter that consume the Domain's seven delivery ports.
- **The Adversary integration.** Architecture §8 names the Adversary as a future read-only Interface client. Sprint-02 ensures the Domain admits a read-only audit adapter via `ReadPorts` narrowing but does not implement Adversary behaviors.
- **Substrate alternatives.** The architecture admits CozoDB, Soufflé, and SQLite recursive-CTE adapters (Architecture §2.4). Sprint-02's design must not preclude swapping; sprint-02 does not implement any non-primary substrate.
- **Cross-cutting adapter implementations.** Default adapters (real clock, sequential-per-category ID, strict consent verification, JSON-file persistence) are listed in Architecture §2.4 and constructed by the Interface at session start. Sprint-02 accepts them as injected dependencies; it does not implement them.

## Key Decisions

The structural decisions inherited from the winning Purist design. Each is named with the alternative considered and the principled reason for the choice.

1. **Verb-as-data with typed `customPostCheck` escape hatch.** Eight operation verbs (per Domain Spec §6.1) are typed `OperationSpec` records in a single `OPERATION_SPECS` registry inside `mutations.js`. Each record carries fields for `consentCategory`, `preconditions` (declarative `QueryPattern[]`), `idShape`, `translate` (returning `{baseFacts, rules, metaFacts}`), `postconditions` (declarative), optional `customPostCheck: (args, ports: ReadPorts) → DomainError | null` (for post-conditions that call policy-module functions rather than reducing to query patterns — e.g., `presentClosingArgument` calling `closurePolicy.triggerGate`), `clearsTwoYes: boolean`, and `resultShape`. One `runOperation(verbName, args, consent, ports: FullPorts)` function (~60 LOC) reads each record and executes Domain Spec §6.1 line-by-line. **Alternative considered:** verb-as-code in plain JS functions per Pragmatist, or verb-as-callback-body via a `runMutation(opName, args, consent, body, ports)` combinator per Conservator. **Why this wins:** the §6.1 pattern lives in one named location rather than being implicit across eight function bodies; the `OperationSpec` validator refuses incomplete records at boot, so drift becomes type-impossible rather than review-dependent; the typed escape hatch honestly admits the irreducible 5% of verbs whose post-conditions are intrinsically procedural without surrendering the data-shape for the symmetric 95%; the architecture's §13.2 OCP-for-Datalog-rules promise extends *upward* from rules to operation verbs as the spec's own deepest structural principle one layer up.

2. **Role-narrowed port bundles enforced at the parameter type.** The bridge constructs four frozen, strictly-nested port bundles: `ReadPorts = {query, explain}`, `WritePorts = {facts, rules, query, explain, tx}`, `ProbePorts = {query, explain, snapshot}`, `FullPorts = WritePorts ∪ ProbePorts ∪ {clock, ids, consent, persist}`. Every Domain function declares which bundle it consumes by its parameter type — renders take `ReadPorts`; counterfactual takes `ProbePorts`; the orchestrator (`runOperation`) takes `FullPorts`; the only file that constructs concrete bundles is `domain-bridge.js`. **Alternative considered:** flat single PortBundle (Innovator round 1; Pragmatist's earlier proposals); accessor methods on bridge returning narrowed objects on request (Pragmatist's final). **Why this wins:** the architecture's §4.1 promise of "a read-only audit adapter dependent only on `IQueryEngine`+`IExplain`" becomes structurally implementable rather than aspirational — a render *literally cannot* call `ports.facts.assertFact` because the function does not have the reference; the contract is visible at the function signature rather than at the first body line or at a call-site accessor invocation; ISP becomes a type-level property (parse-time in TypeScript; runtime-frozen-object enforcement in JS) rather than a code-review convention.

3. **Schema/translation SRP split into two files.** `schema.js` carries `CATEGORY_REGISTRY` of typed `CategoryDescriptor`s with shape metadata, required/optional fields, closed-enum constraints, authority directives, and render-section bindings — pure validation and metadata only, no Engine references. `translation.js` carries the generic translator (one pure function per element category producing `{baseFacts, rules, metaFacts}`) and `RULE_TEMPLATES` (parameterized rule shapes for approval-gated categories). **Alternative considered:** single-file `CATEGORY_DEFS` with internal `// ── SHAPE ──` and `// ── TRANSLATION ──` comment fences (Pragmatist's final). **Why this wins:** Domain Spec §3 (per-category shape requirements) and §2 (the translation table as "the Domain's most important design artifact") are separately specified in the spec — they have different reasons to change (ADR-0004 and ADR-0011 in cascade history were translation-only changes); two files are the SRP-faithful encoding; comment fences in a single file rot when refactored.

4. **Three-phase rule registration with cold-start template stratification.** Phase A: at bridge construction, each policy module's `registerStatic(ports)` defines closure, integrity, lane, and friction-detection rules; stratification (ADR-0013 Part 3) runs at `defineRule` time so cyclic-negation rule sets fail at boot. Phase B: at bridge construction, `translation.registerRuleTemplates(ports)` installs parameterized rule shapes for approval-gated categories (Proposition, Resolution, Concern-coverage, Definition) with placeholder ids; stratification analysis runs on each template. Phase C: when `runOperation` runs `addElement`, the relevant template is instantiated with the new element's id and the concrete rule is defined via `ports.rules.defineRule` inside the open transaction (ADR-0013 read-own-writes makes this visible to subsequent in-tx queries). **Alternative considered:** lazy element-rule registration at first `addElement` per category (Conservator round 1). **Why this wins:** a buggy `ruleShape` for any element category fails at session-init rather than at the first mutation that exercises it; the architecture's fail-fast-at-call-site-that-caused-the-error pattern (Engine Spec §7) is preserved by validating the template *at the place that introduced it* (the bridge) rather than at the first call to `addElement`.

5. **Save inside `runOperation`'s success path, outside the engine transaction, with typed divergence error.** The save-state call lives inside `runOperation`'s try-on-success block but outside the engine `tx.run` boundary because `ITransaction` is engine atomicity, not durability. On save failure, `runOperation` throws `POST_COMMIT_SAVE_FAILED` as a typed Domain error carrying `{engineCommitted: true, cause}`. **Alternative considered:** save at the Interface boundary as a load-call-save wrapper that the Interface owns (Pragmatist's final, citing Architecture §6.2 — "Domain owns the log structure; Interface persists it"). **Why this wins:** Domain Spec §6.1 step 9 puts save in the operation's success ordering, not outside the operation; the Interface layer (sprint-03) gets a one-line MCP adapter per delivery port rather than multi-line load-call-save scaffolding at every tool handler; the divergence at save-failure (engine committed; disk did not) is a real architectural boundary state and the Domain owns its *naming* via a typed error, completing the §11.8 trust-boundary discipline the architecture already names.

6. **Closed-set discipline as a physical module (`tags.js`).** Frozen objects for `INFERENCE_PATTERNS`, `FRICTION_SHAPES`, `FRICTION_DISPOSITIONS`, `ACTION_LABELS`, `WITHDRAWAL_DISPOSITIONS`, `CONSENT_SOURCES`, `ELEMENT_CATEGORIES`, `PHASES`. `assertExhaustive(tag, set)` helper. Every other module imports from `tags.js`; no module re-declares any enum value. **Alternative considered:** inlining enum sets per-category in `schema.js` (the spec's literal §3 framing). **Why this wins:** Architecture §11.11 names closed-set discipline as load-bearing for the channeling architecture's bite ("every accommodation weakens the constraint that makes the channeling work") — a physical home for the discipline makes that property structural rather than convention.

7. **One assembly seam: `domain-bridge.js`.** The bridge is the only file that names a concrete Engine instance and constructs the four port bundles. It takes `{engine, clock, idAllocator, consentVerification, persistenceRepo}` from the Interface at session start, builds the bundles, runs Phase-A and Phase-B rule registration once, and exposes the seven delivery-port surfaces as one-line facades over `runOperation` or render/query calls. The bridge holds no mutable session state. **Alternative considered:** assembly distributed across modules with implicit ambient context. **Why this wins:** substrate swap (Architecture §11.4 cost) is a one-file change; the architecture's DIP property (§13.5) becomes structural — every Domain module imports port *type signatures*, never Engine implementation classes.

## Constraints

- **Architectural cascade is normative.** All decisions inherited from `03-architecture.md`, `04-engine-spec.md`, `05-domain-spec.md`, and the 19 ADRs are binding. No structural choice may contradict the cascade; only structural realizations of the cascade's commitments.
- **Six substrate ports only.** Engine consumption goes through `IFactStore`, `IRuleStore`, `IQueryEngine`, `ISnapshotRestore`, `IExplain`, `ITransaction` (per ADR-0012 as amended by ADR-0013, which dropped `IMaterializer` and specified read-own-writes transaction visibility). No Domain module may reach Engine internals.
- **Seven delivery ports only.** Interface consumption enters through `IElementMutation`, `IRatification`, `IFrictionManagement`, `IDefinitionManagement`, `IClosureSurface`, `IRenderSurface`, `IQuerySurface` (per Architecture §4.2). The Domain exposes only these.
- **Four cross-cutting adapters, injection only.** `IClock`, `IIDAllocator`, `IConsentVerification`, `IPersistenceRepository` (per Architecture §4.3, ADR-0009). No Domain module reads time directly, generates IDs inline, validates consent shape, or touches files.
- **Forward-solve paradigm preserved (ADR-0007).** Propositions and Resolutions are derivable rules with approval as a body literal (ADR-0003). No procedural alternative — verbs assert facts and define rules; derivation is the engine's job.
- **Closed-set tags stay closed (Architecture §11.11).** No string-typed alternatives where a closed-set enum exists. Every new tag value goes through an explicit review.
- **Stratified-Datalog dialect (Engine Spec §2).** Domain rule definitions must be valid in the stratified-Datalog family. ADR-0013 Part 3 specifies stratification check at `defineRule` time.
- **LOC budget ~2,050.** Per Domain Spec §13's 1,500–2,500 range. Eleven files plus bridge. Individual module ceilings: `schema.js` ~280; `translation.js` ~270; `closure-policy.js` ~240; `mutations.js` ~280; `render.js` ~340; others smaller.
- **JavaScript (not TypeScript) for the primary implementation.** The cascade does not commit to TypeScript; the role-narrowed bundles enforce ISP at runtime via frozen objects rather than at parse-time via type signatures. (A future migration to TypeScript would tighten the discipline; the design admits this without rewriting modules.)
- **No I/O, no global state, no ambient context.** Every Domain module is a pure function over its declared port bundle plus the call's arguments. Per Domain Spec §13 closing paragraph.

## Acceptance Criteria

Observable, testable conditions for the structural design's correctness. These criteria are evaluated against the produced spec by `design-specify` and against the produced implementation by `plan-build` / `execute-write` downstream.

- **Module-count and file-shape match.** Eleven files plus `domain-bridge.js` exist with the names `tags.js`, `schema.js`, `translation.js`, `authority.js`, `lifecycle.js`, `closure-policy.js`, `friction-policy.js`, `restructuring.js`, `render.js`, `counterfactual.js`, `mutations.js`. Total Domain LOC inside the 1,500–2,500 envelope.
- **Each module declares its port-bundle dependency at the function signature level.** Renders take `ReadPorts`; counterfactual takes `ProbePorts`; policy queries take `ReadPorts`; `registerStatic` and `registerRuleTemplates` take only `{rules}`; `runOperation` takes `FullPorts`. A grep across the codebase confirms no Domain module references the concrete Engine instance outside `domain-bridge.js`.
- **`mutations.js` contains one `runOperation` function and one `OPERATION_SPECS` registry.** Eight `OperationSpec` records (`openProof`, `addElement`, `reviseElement`, `withdrawElement`, `ratifyElement`, `manageFriction`, `presentClosingArgument`, `confirmClosureGo`). Each record carries all required fields per the `OperationSpec` type. The `customPostCheck` field appears on at most 3 of 8 records.
- **`runOperation` body matches Domain Spec §6.1 line-by-line.** A test reads the §6.1 step list and asserts that `runOperation`'s control flow visits each step in order with the named branches. The validator refuses to construct an `OperationSpec` missing a required field.
- **Bridge construction triggers stratification check on every defined rule and registered template.** A test that constructs the bridge with a deliberately cyclic-negation rule shape fails at bridge construction, not at first mutation.
- **Render functions structurally cannot mutate state.** A test asserts that no `render.js` function reaches `facts.assertFact`, `rules.defineRule`, or `tx.begin`. The narrowed `ReadPorts` parameter is the structural enforcement.
- **Counterfactual functions restore snapshot on throw.** A test injects a query failure mid-counterfactual; the engine state is bit-equal to the pre-counterfactual snapshot afterward.
- **Save-after-commit surfaces typed divergence on disk failure.** A test that injects an `IPersistenceRepository.saveState` failure after a successful engine commit produces a `POST_COMMIT_SAVE_FAILED` Domain error with `{engineCommitted: true, cause}` payload.
- **The seven delivery-port facade methods on the bridge each invoke `runOperation` or a render/query function with no business logic.** A grep confirms each facade method body is one line plus return.
- **The cross-cutting adapters are injected at bridge construction only.** Grep confirms no Domain module imports a concrete `IClock`, `IIDAllocator`, `IConsentVerification`, or `IPersistenceRepository` implementation outside `domain-bridge.js`.
- **The architecture's payoffs in §10 are mechanically realizable.** A read-only audit adapter that imports only `ReadPorts` from the bridge compiles, runs, and produces correct renders against a populated state. (This is the structural test for ADR-0013's amended ISP discipline.)
- **Sprint-03 (Interface) can consume the Domain via one-line MCP tool adapters.** A representative MCP tool handler for `ratify` is a single line: `return bridge.ratifyElement(args, consent)`. No load-call-save scaffolding at the tool-handler layer.
- **Domain test suite is sized per Domain Spec §12.** Module-level tests against mock substrate adapters; integration tests against the real bridge with a mock or in-memory substrate; no Engine mocking inside Domain tests (the engine is consumed through its real ports).

## Provenance

This brief is the synthesis of a four-pole Cartesian design debate (Conservator / Innovator / Pragmatist / Purist) executed via Claude Code agent-teams with peer-to-peer DMs. The debate ran three elimination cycles (4 → 3 → 2 → 1). The Purist (E pole — structural discipline) won the final head-to-head against Pragmatist (W pole — minimum ceremony) on two decisive axes named by the human evaluator: maintainability for a sole developer six months out (type system substituting for absent peer review), and external customer usability for sprint-03 (load/save inside `runOperation` gives one-line MCP adapters versus multi-line scaffolding at every tool handler). The structurally near-identical Innovator (N pole — leverage) was eliminated first to preserve diversity of remaining positions; Conservator (S pole — spec §13 fidelity) was eliminated second to preserve the cleanest head-to-head between the two genuinely opposed philosophies.

The complete debate transcripts (Round 1 proposals, Round 2 critique-and-revise, Round 3 final designs, and Cycle 2 / Cycle 3 head-to-head outputs from each pole) are available in the session conversation history and represent the proof envelope for the choices captured in Key Decisions above. The eliminated designs survive in this brief as the named alternatives that the winning design's choices were measured against.
